Using the shared LLM

Three ways to reach the backend model, all authenticated with a per-user key you mint on the API Keys page.

Endpoint at a glance

Base URL
{{ llm_base_url }}
Model
{{ llm_model }}
Auth
OpenAI-compatible: Authorization: Bearer <YOUR_KEY>

1. Direct API integration

Point any OpenAI-compatible client or editor extension at the endpoint above. First mint a key on the API Keys page (it's shown once), then drop it into the config below in place of <YOUR_KEY>.

VS Code — Continue extension

Install Continue, then add this model to ~/.continue/config.yaml: 

models:
  - name: seko ({{ llm_model }})
    provider: openai
    model: {{ llm_model }}
    apiBase: {{ llm_base_url }}
    apiKey: <YOUR_KEY>
    roles:
      - chat
      - edit

VS Code — Cline (agentic)

Prefer agentic, multi-file editing? In Cline's settings choose API Provider OpenAI Compatible, set Base URL to {{ llm_base_url }}, API Key to your seko key, and Model ID to {{ llm_model }}. Cline leans on the model's tool-calling, so expect heavier token use than plain chat.

Any language — curl

curl {{ llm_base_url }}/chat/completions \
  -H "Authorization: Bearer $SEKO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model": "{{ llm_model }}", "messages": [{"role": "user", "content": "Hello"}]}'

Any language — OpenAI SDK (Python)

from openai import OpenAI

client = OpenAI(base_url="{{ llm_base_url }}", api_key="<YOUR_KEY>")
resp = client.chat.completions.create(
    model="{{ llm_model }}",
    messages=[{"role": "user", "content": "Hello"}],
)
print(resp.choices[0].message.content)

2. Local workspaces (your own hardware)

Run the same hardened coding workspace image ({{ workspace_image }}) on your own machine, pointed at the shared LLM. The Self-host page mints a key and generates a personalized kit — a .env, a docker-compose.yml, and an installer for your OS (install.sh or install.ps1), shown once.

Prerequisites

  • SSH client — built into macOS, Linux, and Windows 10+ (the ssh command). Used to log into the workspace.
  • An SSH keypair. If you don't have one, run ssh-keygen -t ed25519, then paste the contents of ~/.ssh/id_ed25519.pub on your profile.
  • Docker + Compose v2 (Linux/macOS) or Docker Desktop with the WSL2 backend (Windows).
  1. On Self-host, pick a harness and generate your kit.
  2. Save the files into a folder, then run the installer for your OS (see below).
  3. Connect: ssh dev@localhost -p 2222

Linux / macOS

Needs Docker + Compose v2. In the folder run bash install.sh.

Windows

Install Docker Desktop (WSL2 backend). In PowerShell run powershell -ExecutionPolicy Bypass -File .\install.ps1, then connect with the built-in OpenSSH client: ssh dev@localhost -p 2222. Prefer bash? Run install.sh from WSL2 or Git Bash instead — both target the same Docker Desktop engine.

Everything runs locally except the model calls, which go to {{ llm_base_url }} with your key. You can also repoint LLM_BASE_URL at a model running on your own host.

3. Remote workspaces (hosted, over Tailscale)

Launch an isolated hosted workspace on the server and reach it by SSH over Tailscale. You only need the Tailscale client installed once; after that it's plain ssh, so VS Code Remote-SSH and scp work too.

Prerequisites

  • SSH client — built into macOS, Linux, and Windows 10+ (the ssh command).
  • An SSH keypair. If you don't have one, run ssh-keygen -t ed25519, then paste the contents of ~/.ssh/id_ed25519.pub on your profile.
  • Tailscale — installed once and signed in via an admin invite (one-time setup below). No other ports or hosts are exposed.

One-time Tailscale setup

  1. Ask the admin to send you a Tailscale invite, then install Tailscale and run tailscale up (sign in with the account you were invited as).
  2. That's it — your access is scoped by policy to the workspace SSH ports on the server and nothing else.
  1. Launch a workspace on the Workspaces page.
  2. Copy its exact connect command from that page — it looks like: ssh dev@{{ workspace_ssh_host }} -p <port> (each workspace gets its own port).
  3. To drop straight into the coding harness, append -t pi (or -t omp).

VS Code Remote-SSH

Add a host block to ~/.ssh/config (use the port shown for your workspace), then "Connect to Host" from the Remote-SSH extension: 

Host seko-workspace
  HostName {{ workspace_ssh_host }}
  Port <port>
  User dev
